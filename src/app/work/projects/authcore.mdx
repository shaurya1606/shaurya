## Overview

AuthCore is a production-grade authentication and authorization system designed to support multi-provider login, role-based access control, and secure session handling. The system integrates OAuth providers, credentials-based login, two-factor authentication, and token-based workflows within a scalable layered architecture.

The project emphasizes strong security primitives, serverless database infrastructure, and stateless JWT session management suitable for modern SaaS applications.

---

## Problem Statement

Modern applications require secure authentication with flexible authorization. Many starter implementations lack proper RBAC, token lifecycle management, and scalable architecture. This results in:

- insecure session handling
- weak password storage
- no role-based permission isolation
- absence of two-factor authentication
- monolithic auth logic
- poor token lifecycle management
- limited scalability for serverless deployments

AuthCore addresses these issues through a modular, security-first authentication architecture.

## System Design

AuthCore follows a layered backend architecture separating routing, business logic, data access, and persistence.

```
Client UI
   ↓
Next.js App Router
   ↓
API Route Handlers
   ↓
Service Layer
   ↓
Query Layer (DAL)
   ↓
Drizzle ORM
   ↓
Neon PostgreSQL
```

Design principles:
```text
stateless session management
separation of concerns
secure token lifecycle
layered architecture
serverless database integration
```

**Architecture Overview**

```
User
   ↓
Next.js Frontend
   ↓
Route Handlers
   ↓
Service Layer
   ├── Auth Service
   ├── Token Service
   ├── Email Service
   ↓
Query Layer
   ↓
PostgreSQL Database
```

**Technology Stack**
- Frontend: Next.js 16, React 19, TypeScript, Tailwind CSS, Radix UI, Framer Motion
- Backend: NextAuth (Auth.js v5), JWT session strategy, Edge middleware
- Database: PostgreSQL (Neon serverless), Drizzle ORM
- Security: bcrypt hashing, UUID token generation, crypto-secure OTP, RBAC authorization
- Validation & Forms: Zod, React Hook Form
- Email: Resend API with HTML transactional templates

**Database Design**
- Tables: usersTable, accountsTable, verificationTokensTable, resetPasswordTokensTable, twoFactorTokensTable, twoFactorConfirmationTable
- Entity relationship: User (1) → (N) Accounts, Verification Tokens, Reset Tokens, 2FA Tokens, 2FA Confirmations
- Design decisions: composite primary keys, cascade delete, referential integrity, typed inference via Drizzle ORM

**API Architecture (structured workflow)**
1) POST /api/auth/signup — user registration
2) POST /api/auth/login — login with 2FA
3) POST /api/auth/verify-email — email verification
4) POST /api/auth/reset-password — issue reset token
5) POST /api/auth/new-password — update password with token
6) POST /api/settings — profile update
7) ALL /api/auth/[...nextauth] — Auth.js handler

**Security Architecture**
- Password security: bcrypt hashing (salt rounds = 10), credential protection
- Two-factor auth: 6-digit OTP, crypto-secure generation, 10-minute expiry
- Email verification: UUID v4 tokens, 1-hour expiry, single-use validation
- Password reset: token-gated flow, TTL enforcement, replay protection
- RBAC authorization: USER, ADMIN, SUPER_ADMIN roles

**Middleware Protection**

```
Request
   ↓
Edge Middleware
   ↓
Auth Check
   ↓
Allow / Redirect
```

**Email Workflow**
- Verification email
- Reset password email
- Two-factor OTP email
- Features: HTML + plaintext, reusable templates, async sending, graceful failure handling

**Authentication Flow**

```
User Login
   ↓
Credential / OAuth
   ↓
Password Validation
   ↓
2FA Check
   ↓
JWT Generation
   ↓
Session Established
```

**Authorization Flow**

```
Request
   ↓
JWT Validation
   ↓
Role Extraction
   ↓
Permission Check
   ↓
Route Access Granted
```

**Performance & Scalability**
- Stateless JWT sessions → horizontal scaling
- Serverless PostgreSQL → dynamic resource allocation
- Layered architecture → maintainability
- Edge middleware → low-latency protection
- Type-safe ORM → compile-time safety

**Challenges & Solutions**
- Token lifecycle: enforce single-use tokens with TTL
- OAuth vs credentials: diverged flows in signIn callback
- Session synchronization: refresh JWT payload on validation
- Serverless DB access: use Neon HTTP driver to avoid pooling limits
