## Overview

LetsKraack is a distributed AI coaching assistant designed to provide structured learning, interview preparation, and resume evaluation in a single unified workspace. The platform integrates AI-generated courses, voice-based mock interviews, resume ATS analysis, and a course exploration marketplace. It is implemented as a full-stack Next.js application with a serverless PostgreSQL backend and modular service-based architecture.

The system emphasizes scalable backend design, low-latency AI interactions, and structured data persistence for conversational workflows.

---

## Problem Statement

Interview preparation tools are fragmented across multiple platforms. Learners typically rely on separate systems for courses, mock interviews, resume reviews, and learning resources. This results in disconnected workflows, lack of personalized learning paths, and limited feedback quality.

Existing solutions also suffer from:
- static resume evaluation without job-context awareness
- non-realistic chatbot-based interviews
- slow AI content generation
- lack of persistent learning progress
- monolithic architecture limiting scalability

LetsKraack aims to unify these workflows into a scalable AI-driven preparation platform.

## Database Design

The system uses a normalized relational schema to ensure data integrity and efficient querying.

### Data Model
The database consists of five core tables:
* **`usersTable`**
    Acts as the primary identity store. It handles profile synchronization with Clerk and serves as the root for all relational data ownership.
* **`coursesTable`**
    The repository for AI-generated learning paths. It stores hierarchical course structures, metadata, and JSON-based chapter content.
* **`enrollCoursesTable`**
    A junction table designed to track user interaction. It monitors active enrollments, completion percentages, and individual learning progress.
* **`discussionRoomTable`**
    The persistence layer for the voice interview system. it stores real-time transcripts, session metadata, and the "Sliding Window" conversation context.
* **`resumeAnalysesTable`**
    Dedicated to resume intelligence. It archives ATS compatibility scores, extracted skill sets, and detailed feedback summaries for every uploaded PDF.

### Entity Relationship
```text
User (1) 
  ├── (N) Courses
  ├── (N) Enrollments
  ├── (N) Discussion Rooms
  └── (N) Resume Analyses
```
 ### API Architecture

LetsKraack's backend is powered by stateless REST APIs designed for modularity and high-speed processing.

#### Core Endpoints

* **`/api/generate-course-layout`**
    Initializes the high-level structure and metadata for a new AI course.
* **`/api/generate-course-content`**
    Triggers parallel processing to generate detailed content for individual chapters.
* **`/api/discussion-room`**
    Handles CRUD operations and session state for mock interview environments.
* **`/api/ai-chat`**
    Manages real-time conversational context and streaming responses from Gemini.
* **`/api/ai-feedback`**
    Processes interview transcripts to generate actionable performance metrics.
* **`/api/resume-analysis`**
    Extracts data from uploaded PDFs to provide ATS scoring and job-match feedback.
* **`/api/enroll-course`**
    Manages the relationship between users and their learning progress.
* **`/api/user`**
    Syncs Clerk authentication identities with the PostgreSQL backend.
* **`/api/voice-to-text`**
    Generates temporary authentication tokens for AssemblyAI real-time transcription.

---

> **Design Philosophy:** > The architecture follows a **Stateless API** design. Every request contains the necessary context for processing, allowing the system to scale horizontally across serverless functions without managing complex server-side sessions.

### AI Pipelines & Logic Flows

The core of LetsKraack is built on three specialized pipelines designed for low-latency and high-accuracy feedback.

#### 1. Course Generation Pipeline
```text
[User Input] 
      ↓
[Gemini Layout Generation] ➔ (JSON Structure)
      ↓
[Store Course Metadata] ➔ (Database Persistence)
      ↓
[Parallel Chapter Generation] ➔ (Concurrency Optimization)
      ↓
[Fetch YouTube Videos] ➔ (External API Integration)
      ↓
[Save Final Course Content]
```
Parallel generation reduces total generation latency by ~40%.

#### 2. Voice Interview Flow
```text 
[User Speech] ➔ (Mic Input)
      ↓
[AssemblyAI Transcription] ➔ (STT)
      ↓
[Conversation Context] ➔ (Sliding Window Retrieval)
      ↓
[Gemini Intelligence] ➔ (Response Generation)
      ↓
[AWS Polly TTS] ➔ (Text-to-Speech)
      ↓
[Audio Playback]
```

The system uses sliding window memory (last 4 messages) to maintain context without exceeding token limits.

#### 3. Resume Analysis Flow
```text
[Resume Upload] ➔ (PDF Parsing)
      ↓
[Job Description Input]
      ↓
[AI Structured Evaluation] ➔ (Category Scoring)
      ↓
[Feedback Persistence] ➔ (User Dashboard)

```
Feedback categories include ATS compatibility, tone, content, structure, and skills.

## Challenges & Solutions

Navigating the intersection of real-time voice and high-volume AI generation presented several technical hurdles.

#### AI JSON Parsing
* **The Problem:** Gemini responses often included unwanted Markdown formatting (like ` ```json ` fences), causing `JSON.parse()` to fail.
* **The Solution:** Implemented a robust utility to strip markdown fences and sanitize the raw string before parsing it into the application state.

#### Latency in Course Generation
* **The Problem:** Generating a full course with 10+ chapters sequentially took upwards of 30 seconds, leading to poor UX.
* **The Solution:** Developed a parallelized generation pipeline using `Promise.all()` to trigger chapter content generation and YouTube video fetching concurrently.

#### Voice Synchronization
* **The Problem:** Significant audio lag occurred between user speech and AI playback, breaking the immersion of a "real" interview.
* **The Solution:** Switched to streaming transcription with AssemblyAI and optimized Gemini prompts to generate concise, high-impact responses for faster TTS conversion.

#### Token Usage
* **The Problem:** As mock interviews progressed, the conversation context grew too large, increasing both latency and API costs.
* **The Solution:** Implemented a **Sliding Window Memory** architecture that only retains and sends the last four messages to the AI, maintaining relevance while capping token overhead.
